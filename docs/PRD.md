GradCheck 中文产品需求文档（PRD）
1. Executive Summary
Problem Statement
东南大学本科毕业要求来源分散、规则复杂且执行节点多，学生难以及时、准确地判断自己距离毕业还差哪些任务。当前缺少一个以“能否毕业”为中心、贯通培养方案、课程成绩、非课程要求、推荐和提醒的统一工具。
Proposed Solution
GradCheck 是一款面向东南大学本科生的移动端 Web 应用，以“毕业任务完成进度”为主线，提供培养方案解析、毕业进度判断、绩点计算、教务信息抓取推荐、选课方案推荐、体育与提醒、换课/组队发帖和毕业指南礼包等能力。系统通过结构化毕业要求、用户进度记录和外部资讯抓取，帮助学生持续了解毕业风险并获得下一步行动建议。
2. 背景与问题
东南大学本科毕业要求分散在培养方案、教务系统、第二课堂、体育制度、志愿活动、劳育课程、学院特色要求等多个来源中。学生需要自行理解规则、手动记录完成情况，并在选课、讲座、实践、跑操、考试、心得提交等节点反复确认信息。现状存在以下问题：
• 培养方案 PDF 难以结构化理解，必修课、选修组别、最低修读门数、学分要求容易漏看或误判。
• 学生无法实时知道“自己还差什么才能毕业”，尤其是选修组别、讲座次数、社会实践学分、体育成绩、志愿时长等非课程要求。
• 绩点计算和毕业/保研目标之间缺少连续追踪，学生很难判断后续课程需要达到多少绩点。
• 教务处、第二课堂、活动通知等信息分散，学生容易错过可补齐要求的讲座、比赛、实践、劳育课、跑操加次数活动。
• 选课系统与培养方案脱节，学生难以生成兼顾毕业要求、时间冲突和个人偏好的选课方案。
• 换课、竞赛组队、社会实践组队主要依靠群聊刷屏，信息不结构化且难以维护状态。
3. User Experience & Functionality
User Personas
• 普通本科生：希望知道自己是否满足毕业条件、还缺哪些课程/讲座/实践/体育/志愿等要求，并及时收到提醒。
• 有保研或特殊培养目标的学生：需要自定义更高 GPA 或学院特色要求，评估未来课程需要达到的成绩目标。
• 同专业低年级学生：希望复用已解析的培养方案，减少手动整理成本，并获得选课与毕业规划建议。
• 活动/竞赛/社会实践参与者：希望找到组队成员、获取适合自己缺口的讲座、竞赛、项目和实践机会。
User Stories
Story 1：培养方案解析与复用
As a 本科生, I want to 上传或选择本专业培养方案 so that 我可以快速建立自己的毕业要求清单。
Acceptance Criteria
• 用户可以上传培养方案 PDF。
• 系统通过 OCR 和 AI 解析识别必修课、选修课、选修组别、组内最低修读数量、学分要求和毕业限制条件。
• 系统必须在保存前展示结构化结果供用户确认或修正。
• 用户可以按学院、专业、年级选择已有培养方案。
• 同一学院、专业、年级已有可用培养方案时，用户无需重复上传。
• 解析后的培养方案必须持久化存储，并记录版本、来源、确认状态和最后更新时间。
Story 2：课程完成进度与毕业判断
As a 本科生, I want to 记录已修课程、选课状态和成绩 so that 我可以知道课程要求是否满足毕业条件。
Acceptance Criteria
• 用户可以标记课程状态：未修、已选、在修、已完成、计划修读。
• 用户可以录入课程成绩、学分和课程类别。
• 系统可以按培养方案判断必修课、选修课、选修组别和总学分是否达标。
• 系统必须明确展示“已完成”“进行中”“存在风险”“未满足”的要求项。
• 当用户修改课程状态或成绩后，毕业进度必须实时重新计算。
• 对缺失课程或未满足组别，系统必须给出具体缺口，而不是只显示“不满足”。
Story 3：绩点计算与目标评估
As a 学生, I want to 计算当前和目标绩点 so that 我可以评估毕业或保研目标是否可达。
Acceptance Criteria
• 系统支持东南大学 4.8 绩点换算规则。
• 用户可以选择计算范围：全部课程、必修课、指定课程集合。
• GPA 结果必须持久化展示，而不是一次性计算后丢失。
• 系统支持默认毕业 GPA 要求 >= 2.0。
• 用户可以自定义更高 GPA 目标，例如保研、吴健雄学院或其他特殊班级要求。
• 系统可以根据剩余课程学分和目标 GPA，估算后续课程需要达到的平均绩点。
Story 4：教务信息抓取与机会推荐
As a 本科生, I want to 接收适合我毕业缺口的讲座、竞赛和项目推荐 so that 我可以及时补齐毕业要求。
Acceptance Criteria
• MVP 阶段接入教务处相关公开信息源，抓取竞赛、课外研学、人文讲座、项目等资讯。
• 系统可以将抓取内容结构化为标题、类型、时间、地点、报名方式、适用对象、可计入要求类别和来源链接。
• 系统根据用户当前缺口推荐讲座、竞赛、项目或其他可补齐要求的机会。
• 推荐必须展示推荐原因，例如“你还差 2 次人文讲座”或“你还缺课外研学成绩”。
• 抓取失败、信息字段缺失或来源不可用时，系统必须标记数据状态，不得将不完整信息伪装为确定结果。
• 用户可以收藏、忽略或标记已参加推荐内容。
Story 5：选课方案推荐
As a 学生, I want to 根据培养方案和个人偏好获得选课建议 so that 我可以更合理地安排课程。
Acceptance Criteria
• 系统结合培养方案剩余要求、课程时间、课程类别和用户偏好生成选课方案。
• 用户可以选择大学期间规划模式：选修课前置集中、均衡分布、后期保留弹性。
• 用户可以选择学期内偏好：集中型、均衡型、上二休五、每日少课等。
• 推荐方案必须避免已知时间冲突。
• 推荐结果必须说明满足了哪些培养方案要求、还剩哪些要求未覆盖。
• 当课程时间或培养方案数据不完整时，系统必须提示不确定性。
Story 6：体育体测、跑操和活动提醒
As a 学生, I want to 记录体测成绩和跑操次数 so that 我可以判断体育相关毕业风险。
Acceptance Criteria
• 用户可以按学年或学期录入体测成绩。
• 系统依据国家体育规范标准和学校毕业证发放要求判断当前体育风险。
• 系统可以估算未来体测需要达到的最低成绩区间。
• 用户可以登记跑操次数。
• 系统可调用天气 API 查询未来晴天趋势，并估算未来可跑操天数。
• 系统抓取或维护跑操次数补充活动信息，并向相关用户推送。
• 对天气预测类结果必须标注预测性质和更新时间。
Story 7：志愿、劳育、实验与考试提醒
As a 学生, I want to 统一管理志愿、劳育、实验和考试提醒 so that 我不会遗漏关键毕业任务。
Acceptance Criteria
• 系统支持记录志愿活动时长和心得提交状态。
• 当志愿活动负责人完成活动信息后，系统应提醒参与学生提交心得体会。
• 系统支持记录劳育理论课时和实践课时完成情况。
• 当劳育课程发布时，系统应及时提醒学生抢课。
• 用户可以自定义登记物理实验课时间段。
• 系统可以在实验开始前按用户配置的提前时间发送提醒。
• 系统可以接入或维护期末考试信息，并提前提醒学生准备和到场。
Story 8：学院特色要求与自定义要求
As a 不同学院的学生, I want to 维护学院特色毕业要求 so that 系统能覆盖非通用毕业条件。
Acceptance Criteria
• 系统内置支持已知学院特色要求，例如计软智学院 IT 讲座、信息学院导师会客厅。
• 用户可以新增自定义要求，包含名称、类型、目标次数/学分/时长、截止时间和备注。
• 自定义要求可以被纳入毕业进度总览。
• 用户可以自行维护自定义要求的完成记录。
• 学院特色要求来源不明确时，系统必须标记为用户自定义或待确认。
Story 9：换课发帖
As a 学生, I want to 发布换课需求 so that 我可以找到愿意交换课程的同学。
Acceptance Criteria
• 用户可以发布换课帖子，包含换出课程、期望换入课程、课程时间、说明和联系方式。
• 系统本身不提供内置聊天功能。
• 帖子必须明确由用户自行填写联系方式。
• 用户可以编辑、关闭或删除自己的帖子。
• 系统支持按课程、学院、时间、状态筛选换课帖子。
Story 10：组队招募
As a 学生, I want to 发布竞赛或社会实践组队信息 so that 我可以找到合适队友。
Acceptance Criteria
• 用户可以发布组队帖子，包含组队目的、项目/竞赛/实践类型、队友要求、当前人数、目标人数和联系方式。
• 帖子必须实时展示队伍仍需人数。
• 用户可以编辑队伍人数和招募状态。
• 系统本身不提供内置聊天功能。
• 系统支持按类型、学院、标签和状态筛选组队帖子。
Story 11：毕业指南隐藏礼包
As a 已完成全部毕业要求的学生, I want to 获得毕业指南文档 so that 我可以顺利完成毕业后续事项。
Acceptance Criteria
• 当系统判断用户所有毕业要求均已完成后，展示隐藏彩蛋入口。
• 彩蛋内容为毕业指南文档或文档入口。
• 如果存在未确认数据，系统不得直接判定“全部完成”，必须提示用户确认。
• 毕业指南可包含毕业流程、材料准备、离校事项、常见问题等内容。
4. AI System Requirements
Tool Requirements
• OCR 服务：用于识别培养方案 PDF 中的文字、表格和课程结构。
• 大模型结构化抽取能力：用于将 OCR 结果转换为课程、组别、学分、必修/选修、毕业条件等结构化数据。
• 规则校验引擎：用于校验 AI 抽取结果是否满足基本逻辑，例如课程学分为正数、选修组别要求不超过组内课程数。
• 人工确认界面：用于用户或管理员确认、修正和发布培养方案解析结果。
• 推荐规则引擎：用于基于用户缺口匹配讲座、竞赛、项目、课程和提醒。
• 信息抓取组件：用于抓取教务公开资讯、活动通知、课程与考试相关信息。
Evaluation Strategy
• 培养方案解析评测集：选取不同学院、专业、年级的培养方案 PDF，构建人工标注答案集。
• 字段级准确率：课程名称、课程代码、学分、课程性质、组别归属、最低修读数量等字段准确率目标 >= 95%。
• 方案级可用率：用户确认后可直接用于毕业判断的解析结果占比目标 >= 85%。
• 毕业判断回归测试：针对典型学生样例构造已完成、缺必修、缺选修组别、缺学分、GPA 不足等场景，确保判断结果和缺口解释一致。
• 推荐命中评估：抽样检查系统推荐是否与用户缺口匹配，推荐原因是否可解释。
• 人工兜底机制：AI 解析置信度低、结构冲突或用户反馈错误时，必须进入人工确认或用户修正流程，不得直接作为确定毕业判断依据。
5. 竞品与差异化
公开可见的同类产品通常分为三类：GPA 与成绩追踪工具、学生日程/课表工具、第二课堂或志愿系统管理平台。GPA 工具如 UniScale、GradeFlow 主要提供课程、成绩、GPA 和目标成绩计算；学生规划工具如 CourseSync、Helium 更关注课表和日程；国内第二课堂系统则偏活动认证、学分统计和学校管理。
GradCheck 的差异化在于：
• 以“能否毕业”为中心，而不是以单点 GPA、课表或活动系统为中心。
• 针对东南大学本科规则进行本地化，包括培养方案、跑操、讲座、劳育、志愿、学院特色要求。
• 支持培养方案 PDF 解析、AI 结构化和同专业复用。
• 把缺口和推荐绑定，例如缺几次讲座就推送相关讲座，缺社会实践学分就推送项目。
• 将大学期间选课策略和学期内排课偏好结合，生成课程规划建议。
6. Technical Specifications
Architecture Overview
GradCheck 采用移动端 Web 前端、后端 API、结构化数据库、AI/OCR 解析服务、信息抓取服务、推荐与提醒服务组成的产品架构。
核心数据流如下：
1. 用户注册或登录后选择学院、专业、年级。
2. 用户选择已有培养方案，或上传 PDF 创建新培养方案。
3. OCR 和 AI 解析服务生成结构化培养方案草稿。
4. 用户或管理员确认后，培养方案进入可复用状态。
5. 用户录入课程、成绩、GPA 目标、体育、志愿、劳育、讲座、实践等完成情况。
6. 规则引擎根据培养方案和自定义要求计算毕业进度。
7. 信息抓取服务定期抓取教务资讯、活动、考试、课程等信息。
8. 推荐引擎结合用户缺口生成讲座、竞赛、项目、选课和活动建议。
9. 提醒服务按用户配置推送实验、考试、活动、心得、劳育、跑操等提醒。
10. 当所有要求满足且数据状态可信时，系统展示毕业指南隐藏礼包。
Integration Points
• 培养方案 PDF 上传：文件上传、OCR 解析、AI 结构化抽取、用户确认。
• 教务公开信息源：讲座、竞赛、项目、课程、考试、跑操活动等资讯抓取；具体 URL、频率和字段以实际可访问页面为准。
• 天气 API：用于晴天趋势查询和跑操可行天数估算；MVP 采用可替换供应商设计，优先选择支持中国城市逐日天气预报的稳定 API。
• 用户认证：MVP 采用基础账号体系，支持手机号/邮箱或校园身份字段绑定；统一身份认证接入作为后续合规评估项。
• 数据库：存储用户、培养方案、课程、成绩、毕业要求、进度、帖子、提醒、抓取资讯和审计记录。
• 通知通道：MVP 必须支持站内通知；微信、短信、邮件或浏览器 Push 作为增强通道，在用户授权和成本评估通过后接入。
Security & Privacy
• 用户成绩、毕业进度、志愿时长、体育成绩等属于敏感个人学习数据，必须最小化采集并仅用于用户授权的毕业进度判断和推荐。
• 系统不得在未经用户授权的情况下公开个人毕业进度、成绩或联系方式。
• 换课和组队帖子中的联系方式由用户主动填写，发布前必须提示其公开可见范围。
• 外部信息抓取必须遵守目标网站 robots、访问频率和服务条款；涉及登录态或个人数据的接入需单独进行合规评估。
• AI 解析结果必须标记来源和确认状态，避免未确认数据影响毕业判断。
• 关键操作需记录审计日志，包括培养方案确认、用户数据修改、帖子发布、推荐生成和提醒发送。
• 数据传输必须使用 HTTPS；敏感字段应按需要进行加密或脱敏展示。
